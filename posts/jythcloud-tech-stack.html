JYTHCloud 技术栈揭秘：我们用什么构建网站和服务

JYTHCloud 从最初的一个简单静态页面，逐步演变成了如今包含主站、团队博客、API 服务在内的一个小型技术生态。这篇文章完整记录了我们当前的技术栈选型、架构思路和踩过的坑，希望能给同样在搞自建站点的同学一些参考。

整体架构：传统多站点 + 统一后端

我们的整体架构并不追求微服务或 Serverless 这些时髦概念，而是采用了更务实的方式——传统多站点 + 统一后端。

具体来说，一台腾讯云服务器上运行着三个站点：

  • 主站（jythcloud.cn） — 团队展示页面，纯静态 HTML
  • 团队博客（wordpress.jythcloud.cn） — WordPress 动态博客
  • API 服务（api.jythcloud.cn） — Flask 提供统一后端能力

所有流量统一经过 Nginx 反向代理，根据域名和路径分发到不同的后端服务。这样做的好处是：运维极度简单，一台机器搞定所有服务；SSL 证书只需在 Nginx 层面统一配置，免去每个服务各自折腾证书的麻烦。

🔑 架构原则：够用就好，不引入不必要的复杂度。对于我们的团队规模和流量，单机多站点架构是最务实的选择。等哪天真的扛不住了再考虑拆分，不提前优化。

前端：纯静态 HTML + CSS + JS

主站的前端选择了最朴素的技术方案——纯静态 HTML + CSS + JavaScript，没有使用 React、Vue 等框架。

选择纯静态的原因很直接：

  • 加载速度极快 — 没有框架运行时，没有构建产物，浏览器直接解析原生 HTML
  • 部署极其简单 — 文件丢到服务器上就能跑，无需 npm install、无需构建步骤
  • 维护成本低 — 不会出现依赖版本冲突、构建失败等问题
  • SEO 友好 — 搜索引擎可以直接抓取完整内容

样式方面使用了统一的 /static/style.css，深色背景（#07070d）配蓝色强调色（#60a5fa），所有页面共享同一套视觉风格。图标字体使用 Font Awesome，通过 CDN 引入。

这种做法对于内容展示型网站来说完全够用。如果将来需要复杂的交互状态管理，可能会引入框架，但目前没有这个必要。

后端：Flask + Nginx 反代

后端选型花了一些时间考虑。最终选择了 Flask + Nginx 反向代理 的组合。

为什么选择 Flask

  • 轻量灵活 — Flask 是一个微型框架，核心功能简单，按需扩展。对于我们的 API 服务来说足够了
  • Python 生态 — 团队对 Python 最熟悉，周边库（SQLAlchemy、Celery、Redis）生态成熟
  • 团队友好 — 在校生团队以 Python 为主要语言，Flask 的学习曲线几乎为零

Nginx 反向代理

Nginx 在我们的架构中承担了多重角色：

  • 流量分发 — 根据 server_name 将请求路由到对应站点
  • SSL 终结 — 统一管理 HTTPS 证书，后端服务无需处理 TLS
  • 静态文件服务 — HTML、CSS、JS、图片等由 Nginx 直接处理，不经过 Flask
  • FastCGI 缓存 — 对 WordPress 站点开启页面缓存，减少 PHP 进程负载

Nginx 的配置大概是这样的思路：

# 主站 — 静态文件直接返回
server {
   listen 443 ssl;
   server_name jythcloud.cn;
   root /www/wwwroot/jythcloud;
   index index.html;
}

# API 服务 — 反向代理到 Flask
server {
   listen 443 ssl;
   server_name api.jythcloud.cn;
   location / {
     proxy_pass http://127.0.0.1:5000;
   }
}

部署与运维

部署和运维方面，我们的目标是：低成本、低维护、高可靠性。毕竟团队人少精力有限，不能花太多时间在运维上。

服务器：腾讯云 4 核 3.6G

服务器用的是腾讯云的轻量应用服务器，配置是 4 核 CPU + 3.6GB 内存。这个配置对于同时运行 Nginx、Flask、WordPress（PHP+MySQL）来说完全够用，价格也在学生团队的承受范围内。

操作系统：OpenCloudOS

选择了腾讯云的 OpenCloudOS 作为操作系统。这是腾讯云基于 CentOS 上游开发的发行版，兼容 CentOS 生态，但更新更及时，而且针对云环境做了优化。选择它的原因很简单：兼容性好、社区活跃、腾讯云原生支持度最佳。

容器化：Docker

Flask API 服务跑在 Docker 容器中。使用 Docker 的好处：

  • 环境隔离 — 依赖不会与宿主机或其他服务冲突
  • 可复现部署 — 有 Dockerfile 就能在任何机器上跑出一模一样的环境
  • 资源限制 — 通过 --memory 参数限制容器内存，防止某个服务耗尽服务器资源

启动 Flask 容器的命令大致如下：

docker run -d \
   --name jythcloud-api \
   --restart unless-stopped \
   --memory 512m \
   -p 127.0.0.1:5000:5000 \
   jythcloud-api:latest

SSL 证书：acme.sh 自动续签

使用 acme.sh 来自动申请和续签 Let's Encrypt SSL 证书。acme.sh 是一个纯 Shell 脚本的 ACME 客户端，安装配置后会在证书到期前自动续签，并 reload Nginx 使新证书生效。全程不需要手动干预。

# 申请证书
acme.sh --issue -d jythcloud.cn \
   --nginx /usr/local/nginx/conf/nginx.conf

# 安装证书（reload Nginx）
acme.sh --install-cert -d jythcloud.cn \
   --reloadcmd "systemctl reload nginx"

每日备份

数据安全不容忽视。我们写了一个备份脚本 backup.py，通过 cron 定时执行，实现每日自动备份：

  • 数据库备份 — WordPress 的 MySQL 数据库完整导出
  • 文件备份 — 站点文件（包括文章、图片、配置）打包归档
  • 远程存储 — 备份文件上传到腾讯云 COS 对象存储，防止服务器故障导致数据丢失

cron 配置：

# 每天凌晨 1 点执行备份
0 1 * * * /usr/bin/python3 /www/wwwroot/jythcloud/backup.py

性能优化

虽然我们的流量不大，但性能优化从项目一开始就纳入了考量。好的架构习惯在流量增长时能省去很多重构工作。

FastCGI 缓存

对于 WordPress 博客站点，开启了 Nginx 的 FastCGI 缓存功能。当一个页面被第一次访问后，Nginx 会将 PHP 生成的页面缓存起来，后续请求直接返回缓存结果，无需再经过 PHP 和 MySQL 处理。

配置核心思路：

# 定义缓存路径和参数
fastcgi_cache_path /var/run/nginx-cache levels=1:2 keys_zone=WORDPRESS:10m inactive=60m;

# 在 location 中启用缓存
location ~ \.php$ {
   fastcgi_cache WORDPRESS;
   fastcgi_cache_valid 200 60m;
   fastcgi_cache_bypass $no_cache;
   add_header X-Cache $upstream_cache_status;
}

开启后 WordPress 的响应时间从平均 200-400ms 降到了 1-5ms（缓存命中时），效果显著。而且对于评论等动态内容，可以通过 cookie 判断登录用户跳过缓存，确保功能正常。

Flask 内存缓存

Flask API 服务中用到了一些外部数据（如大模型 API 的返回结果、配置数据等），这些数据不需要每次都重新获取。我们在应用层实现了一个简单的内存缓存：

from functools import lru_cache
from datetime import timedelta
import time

# 带过期时间的缓存装饰器
def ttl_cache(seconds=300):
   def decorator(func):
     cache = {}
     def wrapper(*args, **kwargs):
       key = (args, tuple(sorted(kwargs.items())))
       if key in cache:
         val, ts = cache[key]
         if time.time() - ts < seconds:
           return val
       result = func(*args, **kwargs)
       cache[key] = (result, time.time())
       return result
   return wrapper
   return decorator

对于频繁请求且允许一定延迟的数据，使用这个缓存装饰器可以将响应时间从秒级降到毫秒级。当然，如果需要跨进程共享缓存（比如多个 Flask worker），还是得上 Redis，目前单容器场景下内存缓存完全够用。

Docker 内存限制

前面提到过，我们通过 Docker 的 --memory 参数限制每个容器的最大内存使用量：

  • Flask API 容器 — 限制 512MB
  • 其他辅助容器 — 视情况设定

这样做的好处是：即使某个服务的代码出现内存泄漏（比如 Python 进程没有释放大对象），也不会拖垮整台服务器。Docker 的 OOM Killer 会帮你杀掉出问题的容器，而其他服务依然正常运行。

此外，还可以设置 --memory-reservation（软限制）和 --memory-swap（交换分区限制）来做更精细的控制。

📊 效果数据：在启用上述所有优化后，主站首页加载时间约 200ms（静态页面，主要是网络传输时间），WordPress 博客缓存命中时约 5ms，Flask API 缓存命中时约 10ms。服务器内存使用率长期稳定在 50% 左右，CPU 空闲率 90%+。对于学生团队的网站来说，这个表现已经绰绰有余。

总结

回顾整个技术栈的选型过程，核心思路就是一句话：选择简单的方案，把复杂留给业务。

纯静态前端 + Flask 后端 + Nginx 反代 + Docker 部署，这套组合没有什么炫酷的新技术，但每一层都经过了实际检验，足够稳定、足够高效。对于预算有限、人力有限的学生团队来说，这或许是最优解。

当然，技术栈不是一成不变的。随着团队发展和业务增长，未来可能会引入更多组件。但无论怎么演进，务实和简洁会是我们一直坚持的原则。
