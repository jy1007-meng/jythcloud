学完大模型 API 调用之后，我一直在想一个问题：大模型只能回答问题，它又不能自己去查资料、执行操作，那有什么用呢？后来了解到 AI Agent（智能体）这个概念，才明白原来可以让大模型"调用工具"。这篇文章记录了我第一次用 LangChain 搭建一个能自动查天气的 Agent 的完整过程。

AI Agent 到底是什么？

用大白话讲，Agent = 大模型 + 工具 + 自主决策。

普通的聊天机器人是这样的：你问一句，它答一句。而 Agent 是这样的：你给它一个任务，它自己决定需要调用什么工具、按什么顺序调用、拿到结果后怎么整合，最后给你一个完整的回答。

举个例子，你问 Agent "北京明天适合出去玩吗？"——它会先调用天气 API 查北京明天的天气，可能还会查一下空气质量，然后综合这些信息给你一个建议。整个过程不需要你一步一步教它，它自己会规划。

🤔 我的理解过程：刚接触这个概念时我觉得很玄乎。后来想明白了——Agent 本质上就是一个循环：大模型生成思考 → 决定调用哪个工具 → 拿到工具返回结果 → 继续思考 → 直到完成任务。就这么简单。

选框架：为什么用 LangChain

市面上 Agent 框架不少，LangChain、AutoGPT、MetaGPT、CrewAI 等等。最终选了 LangChain，原因是：

  • 社区最大，文档相对全。 出问题了 Stack Overflow 上基本能搜到答案。
  • 上手简单。 核心代码不超过 30 行就能跑起来一个简单的 Agent。
  • 工具生态好。 内置了很多工具，比如搜索引擎、计算器、Python REPL，直接 import 就能用。

当然 LangChain 也有缺点，后面我会说。

搭建过程

首先装包：

pip install langchain langchain-openai python-dotenv

第一步：定义工具

Agent 需要"工具"才能干活。我先写了一个简单的天气查询函数作为工具：

# 模拟查天气的函数
def get_weather(city):
    # 实际开发中这里调的是开放天气 API
    weather_data = {
        "北京": "晴，20-28°C，空气质量良",
        "上海": "多云，22-30°C",
        "广州": "阵雨，25-32°C"
    }
    return weather_data.get(city, f"暂无{city}的数据")

第二步：把函数包装成 LangChain 工具

from langchain.tools import tool

@tool
def check_weather(city: str) -> str:
    """查询指定城市的天气情况，输入城市名称"""
    return get_weather(city)

这个 @tool 装饰器是关键——它告诉 LangChain："这是一个工具，它的描述是 xxx，参数是 xxx"。大模型就是根据这个描述来决定什么时候调用这个工具的。

第三步：创建 Agent

from langchain_openai import ChatOpenAI
from langchain.agents import create_tool_calling_agent, AgentExecutor

llm = ChatOpenAI(
    model="deepseek-chat",
    api_key="sk-xxxx",
    base_url="https://api.deepseek.com/v1"
)

agent = create_tool_calling_agent(llm, [check_weather])
agent_executor = AgentExecutor(agent=agent, verbose=True)

result = agent_executor.invoke({"input": "北京明天天气怎么样？适合出去跑步吗？"})
print(result["output"])
💡 第一次跑通的过程：设置 verbose=True 之后，终端会打印 Agent 的"思考过程"。我看到它先判断"用户问的是北京的天气"→调用 check_weather("北京")→拿到"晴，20-28°C"→然后自己推理说"20-28°C 适合跑步"→最后生成完整的回答。整个过程就像在看一个机器人在脑子里自言自语，还挺震撼的。

遇到的问题

搭建过程不是一帆风顺的，记录几个坑：

  • 工具描述要写清楚。 一开始我写的 tool 描述是"查天气"，结果 Agent 总是不调用它。改成"查询指定城市的天气情况，输入城市名称，返回温度和天气状况"之后，调用就准确多了。模型的工具调用能力很大程度上取决于你写的描述够不够清楚。
  • LangChain 版本兼容问题。 LangChain 更新很快，网上搜到的教程可能是旧版本的 API。我安装的是最新版，create_tool_calling_agent 的用法和旧版 agent 初始化方式不一样，对着文档调了半天。
  • Token 消耗比想象的多。 每次 Agent 思考、调用工具、拿到结果再思考，每一步都要消耗 token。一个简单的问答可能比普通对话贵 3-5 倍。开发调试阶段建议用便宜的模型（比如 DeepSeek 就比 GPT 便宜很多）。

对 Agent 的几点感受

做完这个小项目之后，我对 Agent 有了更实际的认识：

  1. Agent 很强，但不可靠。 有时候它会调用错误的工具，或者解析工具返回的结果出错。在实际应用里，必须做好容错和兜底。
  2. 关键在于工具设计。 Agent 的能力上限取决于你给了它什么工具。工具设计得好，Agent 能做很复杂的事情；工具设计得不好，它就是个只会聊天的花瓶。
  3. Debug 有点麻烦。 因为 Agent 的决策过程是不确定的，同样的输入可能每次走不同的路径。设了 verbose=True 能看到思考过程，但还是不好调。
  4. 适合做自动化，不适合做高精度任务。 如果任务需要 100% 准确（比如银行转账），用 Agent 风险很大。但如果只是整理信息、查资料、生成报告这类"做到 80% 就能用"的任务，Agent 非常合适。

下一步我打算给 Agent 多加几个工具，比如文件读写、网页抓取，然后试试能不能让它自动帮我写周报。等做出来了再分享经验。
