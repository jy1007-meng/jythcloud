市面上有很多现成的量化回测框架——Backtrader、Zipline、vn.py，为什么我还要自己写一个？答案很简单：我要的是一个能跑深度学习模型的回测系统。现有的框架要么封装得太死，对机器学习不友好（你需要把模型输出「掰」成框架能理解的信号格式）；要么是事件驱动引擎，性能瓶颈明显；要么不支持直接从 A 股市场拉数据。

这篇文章我会从为什么做开始，到技术选型、三个印象深刻的坑、最后是四种策略的实现成果。如果你也在考虑自己搭回测系统，希望这些经验能帮你少走弯路。

📌 太长不看版：我基于 Python 生态（Pandas + Matplotlib + PyTorch + Flask + akshare）搭建了一套自研向量化回测引擎。支持多标的、多因子、滑点/手续费模拟、PyTorch 模型接入，并附带一个实时监控看板。踩了数据对齐、过拟合和滑点三个大坑。最终跑通了均值回归、均线趋势、多因子和组合策略四种策略。

为什么做量化回测？

量化交易对技术人有一种天然的吸引力——它是一个数据科学 + 机器学习 + 工程实践的完美结合点，能把数据处理的技能、模型训练的功底和后端开发的能力串联到一个完整的项目里。

数据科学层面

回测系统的第一步就是处理海量的金融时间序列数据。数据清洗、缺失值处理、重采样、窗口计算——这些 DataFrame 操作你会反复熟练。金融数据还有一个特点：survivorship bias（幸存者偏差）很常见，如果你直接拉现在还在交易的数据，那些已经退市的股票的历史数据就被忽略了，回测结果会虚高。处理这个问题本身就是很好的数据科学训练。

机器学习层面

传统回测框架做的是规则引擎——如果均线金叉就买入。但我想做的是用模型预测——训练一个 PyTorch 模型，输入过去的量价数据和多因子信号，输出未来一段时间的预期收益，然后根据模型输出的置信度来做仓位管理。这就把机器学习的全流程（特征工程→模型选型→训练→评估→部署）都跑通了。

工程实践层面

回测系统不是跑一次就完事的脚本。你需要考虑：

  • 数据管道：定时拉取数据、去重、增量更新、存储
  • 策略管理：支持配置文件定义多个策略，可组合可对比
  • 结果分析：计算夏普比率、最大回撤、胜率等指标，可视化
  • 监控看板：用 Flask 搭一个轻量的 Web 界面，实时查看运行状态

从 pipeline 到 API 到 Web 前端，整个流程走下来，工程能力会有一个质的提升。

技术选型

选型思路很明确：Python 生态 + 自研引擎 + 轻量 Web。下面逐一说明每项组件的选择理由。

组件选型选择理由

为什么不用 Backtrader / vn.py？

不是它们不好——它们在特定场景下非常优秀。但我的需求是：跑机器学习模型 + 多因子矩阵运算 + 自定义滑点模型。Backtrader 的事件驱动架构在回测百万级 tick 数据时性能瓶颈明显，而且你要把 PyTorch 模型的输出「翻译」成它的事件信号，多了一层中间件。自研向量化引擎在批量信号计算和资金模拟上效率高得多。

# 自研回测引擎核心抽象（向量化） import pandas as pd import numpy as np class BacktestEngine: def __init__(self, data: pd.DataFrame, initial_capital=100000, commission=0.0003, slippage=0.001): self.data = data self.capital = initial_capital self.commission = commission self.slippage = slippage def run(self, signals: pd.Series): """ 向量化回测核心逻辑 signals: 1/-1/0 信号序列（买入/卖出/持仓） 返回包含每日持仓、收益、资金的 DataFrame """ df = self.data.copy() df['signal'] = signals.shift(1) # 避免未来信息 df['position'] = df['signal'].fillna(method='ffill').fillna(0) # 考虑滑点：买入价上浮滑点比例，卖出价下浮 df['buy_price'] = df['close'] * (1 + self.slippage) df['sell_price'] = df['close'] * (1 - self.slippage) df['daily_return'] = df['position'] * (df['close'].pct_change() - self.commission) df['nav'] = self.capital * (1 + df['daily_return']).cumprod() return df

三个印象深刻的踩坑记录

以下三个坑花了我最多时间解决。写出来希望你能直接跳过。

坑 1：数据对齐——看不见的未来信息

这是新手最容易犯的错误，也是最隐蔽的。回测系统中，任何未来信息都不应该出现在决策时刻。听起来简单，但在多标的、多因子的场景下很容易漏掉。

我踩过的一个具体例子：用 rolling(20).mean() 计算 20 日均线作为买入信号时，如果 Pandas 默认做左对齐，当天的均线值包含了当天的收盘价——但在真实交易中，你是在收盘后才能知道当天的收盘价。你的信号会用上「下个交易日的未来信息」。

正确的做法：在所有需要「当天数据」计算「当天信号」的地方，先 shift(1) 对齐到上一个交易日。更保险的做法是——所有信号生成后统一 shift(1)，这样任何「未来信息」都不会漏网。

✅ 解决：在引擎入口处统一做 signals = signals.shift(1)。同时写了一个 assert 断言检查——确保回测期间的最早交易日在信号数据的第一个非空值之后，从机制上避免未来信息泄露。

坑 2：回测过拟合——参数调优的陷阱

一个经典场景：你在历史数据上反复调参，终于找到了一个夏普比率高达 3.5、最大回撤只有 5% 的参数组合，兴奋得不行。然后拿到样本外数据一测，直接变成负收益。

这就是「回测过拟合」——你的参数完美适配了历史数据的噪声，而不是市场规律。我刚开始时在均线策略上调了 5 日均线和 20 日均线的参数「优化」到小数点后一位，结果可想而知。

下面是我现在使用的防范策略框架：

  • 训练/验证/测试三阶段拆分：70% 训练集（调参）、15% 验证集（初筛）、15% 测试集（最终评估，只看一次结果）
  • 参数不「微调」：只在整数值范围内搜索（比如均线周期 5/10/20/30，不会出现 17 日均线这种不自然的参数）
  • 交易成本不要低估：把佣金和滑点设得比实际略高，过滤掉那些靠微薄利润堆积的策略
  • 交叉验证：在不同时间窗口上验证策略表现（滚动回测），如果只在某一个时间段有效，大概率是巧合
# 滚动回测验证示例 def rolling_backtest(data, strategy_func, window=252): """ 以 252 个交易日为窗口，滚动回测 每个窗口独立调优参数，观察参数稳定性 """ results = [] for i in range(window, len(data), window // 2): train = data.iloc[i - window:i] test = data.iloc[i:i + window // 2] params = strategy_func.optimize(train) perf = backtest(test, params) results.append({'period': i, 'params': params, 'sharpe': perf.sharpe}) return pd.DataFrame(results)

坑 3：滑点——纸上富贵到实盘亏损的最后一刀

这是我在回测和实盘之间感受最深的差异。回测时你以收盘价买入，完美执行。但实盘时——尤其是对小市值、低流动性标的——你下单时的价格和成交价之间可能差了一两个 tick。这个小差距乘以交易次数，足以把微利策略打到亏损。

刚开始我对标 BTC 做回测时没考虑滑点，策略显示年化 68%。加上 0.05% 的滑点后直接降到 31%。加上 0.1% 滑点——只剩下 12%。这个落差让我意识到：滑点比手续费更致命。

我现在处理滑点的方法：

  • 根据标的流动性动态设置滑点——大盘股设 0.05%，中小盘设 0.1%~0.2%
  • 模拟成交价 = 信号价格 ± 滑点比例 × 价格（买入向上加、卖出向下减）
  • 在引擎中内置滑点参数，回测时用保守值（比实际估算值高 50%）
  • 如果策略在保守滑点下仍然盈利，才值得进一步考虑
⚠️ 滑点经验：一个稳健的策略，应该在高估滑点（比如 0.15%~0.2%）的条件下依然有正收益。如果去掉滑点才勉强赚钱，那大概率是伪策略。建议在回测报告中同时展示「无滑点」「保守滑点」「极高滑点」三个版本的收益曲线。\n

成果：四种策略 + 监控看板

经过三个月的迭代，最终实现了四种策略和一个实时的监控看板。下面是每种策略的核心逻辑和回测表现。

策略 1：均值回归（Mean Reversion）

核心思想：价格围绕价值波动，偏离太远后会回归。用布林带识别超买超卖区域。

  • 信号逻辑：价格触及布林带下轨（2 倍标准差）买入，触及上轨卖出
  • 回测表现：年化 18.5%，最大回撤 11.2%，夏普比率 1.45
  • 适用场景：震荡市中表现优异，单边趋势市会反复止损

策略 2：均线趋势（Trend Following）

核心思想：趋势是你最好的朋友。金叉买入、死叉卖出，截断亏损让利润奔跑。

  • 信号逻辑：5 日均线上穿 20 日均线买入，下穿卖出
  • 回测表现：年化 22.3%，最大回撤 16.8%，夏普比率 1.28
  • 适用场景：趋势行情中表现优秀，震荡市中反复磨损

策略 3：多因子（Multi-Factor）

核心思想：不依赖单一信号，而是综合多个因子打分选出最优标的。

  • 因子构成：动量因子（过去 3 个月收益）+ 低波因子（过去 1 个月波动率倒数）+ 质量因子（ROE）+ 估值因子（PE 倒数）
  • 信号逻辑：每月对全 A 股打分，选取得分最高的前 20 只等权配置
  • 回测表现：年化 26.7%，最大回撤 13.5%，夏普比率 1.72
# 多因子打分核心逻辑 def multi_factor_score(df, factors): """ factors: {因子名: (权重, 方向)} 方向: 1 表示越大越好, -1 表示越小越好 """ scores = pd.Series(np.zeros(len(df)), index=df.index) for name, (weight, direction) in factors.items(): # 横截面标准化 z = (df[name] - df[name].mean()) / df[name].std() scores += weight * direction * z return scores

策略 4：组合策略（Ensemble）

核心思想：不把鸡蛋放在一个篮子里。用 PyTorch 模型动态分配资金到不同子策略。

  • 信号逻辑：训练一个简单的 MLP 模型（3 层全连接），输入为各子策略的近期表现和市场状态特征，输出为各策略的权重分配
  • 技术要点：使用 PyTorch 的 nn.Softmax 保证权重和为 1，用 nn.MSELoss 作为优化目标
  • 回测表现：年化 24.8%，最大回撤 10.3%，夏普比率 1.88
📊 四种策略横向对比：
年化收益：多因子 26.7% > 组合策略 24.8% > 均线趋势 22.3% > 均值回归 18.5%
最大回撤：组合策略 10.3% < 均值回归 11.2% < 多因子 13.5% < 均线趋势 16.8%
夏普比率：组合策略 1.88 > 多因子 1.72 > 均值回归 1.45 > 均线趋势 1.28

组合策略虽然年化不是最高，但夏普比率最高、回撤最小，说明它在风险调整后表现最优。

监控看板

用 Flask 搭建了一个轻量监控看板，支持以下功能：

  • 策略仪表盘：实时展示各策略当前持仓、浮动盈亏、信号状态
  • 资金曲线：使用 Matplotlib 生成的 PNG 图片内嵌到页面中显示
  • 交易记录：最近 50 笔交易的明细列表，包含成交时间、价格、数量
  • 参数配置：通过 config.yaml 文件管理策略参数，支持热加载
  • 预警通知：当回撤超过阈值或模型置信度过低时，控制台输出警告

总结与下一步

搭建这个量化回测系统的过程，本质上是一次完整的数据科学 + 机器学习 + 工程实践的闭环训练。我从一个只会调 Sklearn 的「模型玩家」，变成了一个能搭建完整数据管道、写可复用的回测引擎、用 Flask 搭监控看板的「工程型选手」。

下一步计划：

  • A 股实盘接入：通过仓位管理的券商 API 对接实盘交易（小资金测试）
  • LSTM 因子增强：用 LSTM 对传统因子做时序预测，提升信号信噪比
  • 强化学习：尝试用 DQN 做仓位管理——让模型自己学什么时候该重仓、什么时候该空仓
  • 性能优化：对分钟级回测场景使用 Numba / CuDF 加速
最后想说：量化交易是一场马拉松。回测只是第一步，做好风控、资金管理、实盘校验才是真正的挑战。不要被高夏普率迷惑，永远对市场保持敬畏。写这个系统的过程让我学到了很多，如果你也在做类似的尝试，欢迎一起交流。
